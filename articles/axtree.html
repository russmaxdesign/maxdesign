How Chrome builds the accessibility tree - start to finish

Published: 7 December 2025

Note: The following steps represent my understanding of how Chrome constructs its accessibility tree. This is based on information from Chromium’s open-source codebase. Source references are listed at the end.

We’re often told that the browser builds its accessibility tree ‘from the DOM,’ but what does that actually involve?

Let’s look at how Chrome builds its accessibility tree, from start to finish.

1. Identify which DOM nodes are “interesting” for accessibility

Chrome walks the DOM and filters out anything that has no semantic or interactive value — such as hidden elements, inert content, non-semantic containers, and visual-only wrappers.

2. Compute the AXRole for each element

Blink determines the accessibility role using native semantics first, then ARIA roles when valid, and finally fallback mappings when ARIA is incorrect or unsupported.

3. Compute the Accessible Name + Description

Chrome runs the full Accessible Name and Description Computation algorithm using aria-labelledby, aria-label, native naming rules, and text content.

4. Build the AXNode for this DOM node

For every exposed element, Blink creates an AXNode containing its role, name, description, states, value, and other accessibility metadata.

5. Build the AX Tree Hierarchy (Parent/Child Relationships)

Chrome constructs parent/child relationships using DOM structure, flattening rules, ARIA re-parenting (aria-owns), and virtual nodes for controls like lists, tables, and form elements.

6. Compute Relationships

Blink resolves all semantic links: labelledby, describedby, controls, active-descendant, cell/table relationships, and other references.

7. Compute States and Properties

Chrome computes states such as focus, checked, expanded, disabled, busy, required, and selected, and applies all ARIA states and properties that affect accessibility.

8. Compute Bounding Boxes + Layout-dependent attributes

Chrome retrieves the onscreen position and size of each AXNode based on layout, transforms, scrolling, and composited layers. This step allows hit-testing, virtual cursor navigation, and spatial reasoning for ATs.

9. Finalise the AXTree snapshot

Blink assembles a stable representation of the current accessibility tree, caches computed values, and marks any nodes that have changed since the previous snapshot.

Note: This “what changed?” information is crucial — it is exactly how Blink determines whether an AXEvent needs to fire. It is the differences between the old and new snapshots that trigger accessibility events.

10. Generate AXEvents based on tree changes

Chrome compares old and new AXNode data and fires events (e.g., name changed, value changed, focus changed, live region update) only where actual accessibility-relevant changes occurred.

11. Deliver the AXTree + AXEvents to the OS Accessibility API

Blink maps its internal AXNodes to the platform’s accessibility objects (AX API, UIA, AT-SPI) and updates the platform tree.

12. OS API delivers events → Assistive Technology receives them

The operating system hands these events to the screen reader, which then interprets them and decides what — and when — to speak.

And all of this happens in the blink of an eye (pun intended).

Modern browsers run through all these steps continuously, incrementally, and with a lot of optimisation — often within microseconds.

Sources

Blink Accessibility Core: ax_object.cc

What it covers:

  • Determines which DOM nodes are exposed
  • Computes roles, names, descriptions, states, relationships, value computation, etc.
  • Contains the logic that turns DOM nodes into Blink’s internal AXObjects.

AX Tree Serialization: ax_tree_serializer.h

What it covers:

  • Maintains a “client-known tree”
  • Computes differences between snapshots of the AXTree
  • Generates accessibility events (AXEvents)

AXNode Structure: ax_node.h

What it covers:

  • Defines the actual AXNode objects
  • Holds computed properties for name, role, states, value
  • Defines parent/child relationships
  • Is the structure that gets serialized
