What does “hidden” actually mean in Accessible Name computation?

Published: 14 December 2025

The Accessible Name and Description Computation 1.2 explains how browsers decide which parts of a page can be used to create accessible names and descriptions.

One important part of this specification defines what it means for something to be considered “hidden” during that calculation.

In other words: “Which elements are ignored, and which can still be used to form an accessible name”

The steps below are plain English translations of the specification, so you can clearly see which elements are treated as hidden and which ones still count toward the accessible name.

There are four categories defined in the specification:

1. Hidden via CSS

This includes: display:none, visibility:hidden, visibility:collapse, content-visibility:hidden.

These properties ARE considered hidden.

These elements are not rendered, do not take up space, and do not appear in the accessibility tree.

Verdict: They cannot contribute to accessible names or descriptions.

2. Hidden via SVG or visual transparency

This includes: opacity:0, filter:opacity(0%), or similar SVG transparency techniques.

These techniques ARE NOT considered hidden.

Even though the element is visually invisible, it is still present in the layout, in the DOM, in the accessibility tree, and as selectable or copyable text.

Verdict: They can contribute to accessible names and descriptions.

3. Hidden via ARIA

This includes: aria-hidden="true".

This attribute IS considered hidden.

The author has explicitly instructed the browser to exclude the element from accessibility APIs. It is removed from the accessibility tree.

Verdict: It cannot contribute to accessible names or descriptions.

4. Positioned off-screen or visually covered

These elements remain in the accessibility tree even if they are not visible.

These elements ARE NOT considered hidden.

If an element is moved far off-screen or visually obscured by another element, it still exists in the DOM and in the accessibility tree. This includes patterns like “visually hidden but accessible” text used for contextual labelling.

Verdict: They can contribute to accessible names and descriptions.

Keep in mind

Hidden content is normally ignored in accessible name and description computation. However, aria-labelledby and aria-describedby can reference hidden elements and still use their text to create accessible names or descriptions for other elements.

In the following example, the paragraph is hidden, but it can still provide the accessible name for the <nav> element. The <nav> will have an accessible name of "Hello world".

<nav aria-labelledby="aaa">
  ...
</nav>
<p id="aaa" hidden>Hello world</p>
