What is an AXEvent?

Published: 5 December 2025

An AXEvent is a notification generated by the browser and sent to assistive technologies when something meaningful changes.

This is like the browser telling the screen reader:

“Something just changed that you may want to know about!”

Why AXEvents exist

These events help assistive technologies stay in sync with what’s happening on the screen.

Screen readers do not constantly scan the entire interface looking for changes. That would be slow and inefficient.

Instead, browsers only send AXEvents when something important happens.

This keeps assistive technologies fast and responsive. It also ensures that they aren’t overwhelmed with announcements.

Where AXEvents come from

AXEvents don’t come directly from your HTML or CSS. Instead, they come from the accessibility tree (AXTree).

The AXTree is the browser’s dedicated accessibility structure. It is not a direct copy of the DOM. For example, a simple <div> may be ignored entirely, while a <select> may create multiple accessibility nodes.

Whenever the DOM changes in a meaningful way — even without JavaScript — the browser recomputes the AXTree.

If something important changes (name, value, focus, state, or role), the browser generates one or more AXEvents.

Examples of AXEvents

Here are some simple examples of what can trigger AXEvents:

Changes to focus

  • Moving focus to a button.
  • Tabbing into a form field.
  • Using arrow keys inside a menu.

Changes to value

  • Typing into an <input>.
  • Choosing an option in a dropdown.
  • Adjusting a slider.

Changes to state

  • Opening or closing a disclosure widget.
  • Checking a checkbox or radio button.
  • Showing or hiding an error message.
  • Enabling or disabling a button.

Changes to name or description

  • Updating a label.
  • Updating a live region.

Where AXEvents go

Once the browser generates an AXEvent, it sends it to the platform accessibility API:

  • AX API: macOS and iOS
  • UIA: Windows
  • ATK/AT-SPI: Linux

The operating system then delivers these AXEvents to assistive technologies such as VoiceOver, NVDA, and JAWS.

What screen readers do with AXEvents

Screen readers do not announce everything they receive. They use their own logic and user settings.

A screen reader may:

  • Announce the change immediately.
  • Queue or delay it.
  • Merge it with other events.
  • Interrupt what’s currently speaking.
  • Ignore it entirely.

This is why different screen readers may behave differently even when receiving the same AXEvent.

Summary

Understanding AXEvents helps explain:

  • Why some changes are announced and others are not.
  • Why announcements differ across browsers and platforms.
  • Why “invisible” DOM changes affect screen reader output.
  • Why live regions behave inconsistently.

AXEvents are the messages that keep the browser and assistive technologies in sync.
