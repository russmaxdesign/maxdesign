What are DOM mutations?

Published: 5 December 2025

The DOM is a tree-like structure representing all elements in the document. It is a live, reactive browser data structure, not a passive copy of your HTML.

DOM mutations are any changes to this tree. These mutations can happen with or without JavaScript.

DOM mutations include changes to structure (adding or removing nodes) and changes to attributes (like value, open, or aria-expanded).

The browser constantly modifies the DOM to reflect:

  • state changes
  • user actions
  • validation results
  • interactive behaviour
  • internal UI logic
  • accessibility behaviours in rare cases
  • parsing corrections
  • autofill
  • spellchecking behaviour

These changes happen because many HTML elements have built-in behaviour, and the browser must update the DOM to keep the element’s internal state, visual UI, and accessibility information in sync.

None of this requires JavaScript.

Here are some quick examples you can try yourself:

  1. When you type into an <input>, the browser updates the element’s value property, which counts as a DOM mutation even though you didn’t change the HTML.
  2. When you select an option in a <select>, the DOM automatically updates which option is selected.
  3. When a <details> element opens, the browser sets the open attribute on the <details> element automatically.

JavaScript and the DOM

JavaScript can also be used for DOM mutations. When your script runs, it can directly change the structure, text, or attributes of elements — and every one of these changes counts as a DOM mutation.

Typical JavaScript-driven mutations include:

  • Adding nodes
  • Removing nodes
  • Updating attributes (including ARIA)
  • Changing text
  • Changing internal state (“internal state” refers to properties the browser tracks for an element, such as whether a <details> element is open or whether a form control is valid.)

These are sometimes called “JavaScript-driven DOM mutations”, but they are not really a special category — just another way the DOM changes.

Why DOM mutations matter for accessibility

Every DOM mutation requires the browser to update the AXTree (accessibility tree), the internal structure used by screen readers and other assistive technologies.

Whenever the DOM changes:

  • The browser updates the AXTree.
  • It compares the old and new tree.
  • It fires AXEvents 1 for anything meaningful that changed such as:
    • Name changes
    • Value changes
    • Expanded/collapsed state
    • Focus changes
    • Role changes
    • Live region updates

These AXEvents then travel through the operating system’s accessibility API to the screen reader, which interprets them and decides whether, when, and how to announce the change.

Not all AXEvents result in announcements — screen readers may merge, delay, or ignore certain events based on their behaviour and user settings.

If we return to our previous three examples, they all push information into the AXTree:

  • The <input> value is exposed as the control’s value
  • The <select> option is exposed as the element’s accessible value
  • The <summary> open/closed state is mapped to accessibility state

The browser does not store a separate copy of your HTML — it continuously recomputes accessibility information from the live DOM.

Why this matters

DOM mutations directly influence:

  • What screen readers announce
  • How quickly announcements occur
  • Which events interrupt others
  • Whether changes are even detectable

These effects come from how mutations trigger AXTree updates and AXEvents.

Even without JavaScript, DOM mutations still drive accessibility updates.

Wrapping up

  • The DOM is not static.
  • The browser mutates it frequently without JavaScript.
  • JavaScript can also be used to mutate the DOM.
  • All DOM mutations flow into the AXTree.
  • Accessibility updates, announcements, and timing are shaped by these mutations.
